FindRE User Guide

AI-powered real estate discovery. Describe your ideal home in plain English — FindRE does the rest.

v1.0 · 2026-06-27 React 19 + FastAPI Ollama Local LLM

Overview

FindRE combines a conversational AI intake agent with live Zillow data and a local large language model to surface properties that a keyword search would miss. Instead of filtering by checkboxes, you describe your situation — the app extracts your criteria, searches Zillow in real time, scores every result on three axes, and writes a personalized explanation for each property.

Everything runs locally. The LLM (Ollama + qwen3.6:35b) never sends your data to a third-party AI service. The only external call is to the Zillow RapidAPI — and even that is optional; demo mode ships with seeded San Diego data.

What FindRE is good at: finding properties that exactly or partially fit a nuanced set of criteria, identifying undervalued listings, and surfacing "hidden gems" — properties that have sat on the market long enough for sellers to be motivated.

What it is not: a replacement for a real estate agent. Lot sizes, HOA rules, and disclosure items must be independently verified. Use the AI explanations as a starting point, not a substitute for due diligence.

Installation

Prerequisites

  • Python 3.11+ with uv
  • Node.js 18+
  • Ollama running locally
  • RapidAPI key (optional — demo mode works without it)

Steps

1

Pull the LLM model — this is a one-time ~20 GB download. Start it now so it's ready by the time you finish setup.

ollama pull qwen3.6:35b
2

Install backend dependencies

cd backend
uv sync
3

Install frontend dependencies

cd frontend
npm install
4

Configure environment (optional — required only for live Zillow data)

cp .env.example .env
# set ZILLOW_API_KEY=your_key_here
5

Start both servers

# Terminal 1
cd backend && uv run python main.py

# Terminal 2
cd frontend && npm run dev

Open http://localhost:5173

Your First Search

The chat panel on the left is where you describe what you want. FindRE needs two things to start a search: a location (city, neighborhood, or zip code) and a budget. Everything else is optional.

You
3 bed house in San Diego, budget under $800k, must have a garage
FindRE
Got it — searching San Diego for 3+ bed single-family homes under $800k with a garage...

That's it. FindRE confirms the capture and searches immediately — there's no "shall I proceed?" confirmation.

Tip: You can give more information across multiple messages. Say the location first, then add budget, then mention must-haves. FindRE merges everything across the conversation.

More example queries

You
92120 zip, $600k–$900k budget, big backyard at least 6000 sqft, not a condo or apartment, no HOA if possible
You
Looking for a fixer-upper near Chula Vista. Single family only, private lot. Max $750k. I don't need lots of bedrooms — 2 is fine — but I want space to build an ADU.
You
La Mesa CA, 3-4 beds, $850k max, pool or big yard preferred, deal breaker is shared driveway

Note: If Ollama is offline or unreachable, FindRE falls back to a rule-based intake that asks for location and budget. The scoring and explanation pipeline continue to work without Ollama (rule-based fallback mode). The status bar shows AI Offline when Ollama isn't detected.

The Scoring System

Every property receives three independent scores from 0–100, then a composite weighted average.

Match score

Match
88

How well the property satisfies your stated criteria. Deductions for: over budget (up to −60), missing bedrooms (−25 per bed), missing bathrooms (−15 per bath), under minimum sqft (up to −40), under minimum lot size (up to −40), missing must-haves (−15 each), deal-breaker present (−40 each). Bonus for being well under budget (+10 max).

Value score

Value
72

How the price-per-sqft compares to the neighborhood median within the result set. A property 30%+ below neighborhood average scores ~95; at the median it scores ~55; 30%+ above scores ~20.

Opportunity score

Opp
55

How motivated the seller appears. Points for: 30+ DOM (+10), 60+ DOM (+25), 90+ DOM (+35), 1 price reduction (+15), 2+ price reductions (+25), well under budget (+10), nice-to-haves present (+5 each).

Composite score

A weighted average: composite = (match × w_match + value × w_value + opportunity × w_opp) / 100

Default weights are 40 / 35 / 25. Adjust them with the sliders in the bottom bar — results re-rank in real time.

Property Categories

After scoring, each property is assigned a category that drives the color of its map pin and card badge.

Category Color Criteria
Exact Green Match ≥ 65 and composite ≥ 55. This property checks your core boxes.
Related Blue Match ≥ 40, or value ≥ 65 and composite ≥ 50. Worth considering — some criteria met.
Gem Amber Opportunity ≥ 65 with 60+ DOM, or 2+ price reductions with value ≥ 65. Low match but exceptional signal — seller motivated, possible deal.

Use the category filter tabs at the top of the results list to show only one type. The count in each tab updates live. Gems in particular are worth reviewing even when they're outside your strict criteria — they often represent the best negotiating opportunities.

Hard Filters

Before scoring, FindRE drops properties that definitively fail a hard requirement. Two hard filters are applied:

Private lot / single-family filter

Triggered when you say anything like "not shared", "private lot", "no apartments", "single family only", "no tenants". When active, properties with type multi_family or condo are dropped before scoring.

This matters because Zillow reports the parcel lot size for condos and multi-family units. A 6-unit building on a 20,000 sqft parcel shows 20,000 sqft per unit — a misleading number if you want a private yard.

Minimum lot size filter

When you specify lot_size_min and Zillow has confirmed lot size data, properties with a known lot size below your minimum are dropped. Properties with no listed lot size are not dropped — they pass through but receive a scoring penalty and an explanation flag saying "lot size not listed — verify before scheduling a showing."

Important: Lot sizes reported by Zillow are not always accurate or complete. The hard filter only excludes properties where the data is confirmed to be too small. Always verify lot size directly with the listing agent or county records before scheduling a showing.

AI Explanations

Every property card includes a three-part explanation:

  • Why it matches — how it satisfies your specific criteria
  • Why it's interesting — value signals, days on market, price history
  • Tradeoffs — what it doesn't meet, what to verify

Plus three highlight bullets — short one-liners summarizing the most important points.

The #1 result (highest composite score) gets a full AI-written explanation from qwen3.6:35b. All other results use a fast rule-based explanation generator that produces consistent, accurate output without waiting for the LLM. This keeps total search time under ~35 seconds even with a large LLM.

If Ollama is offline, all results use rule-based explanations — you won't notice a UI difference.

Map View

The right panel shows all results on a CartoDB Dark Matter base map. Pin colors match the category badges:

  • Green pins — Exact matches
  • Blue pins — Related properties
  • Amber pins — Gems (opportunity plays)

Click any pin to open a popup with address, price, and composite score. The popup has a Details button that scrolls the left panel to that property's card and expands the explanation.

A dashed circle on the map shows the search radius from the geocoded center point.

Geocoding: US zip codes are always looked up as "zip, USA" first to avoid false matches with international postal codes that share the same digits.

Score Weights

The bottom bar shows three sliders — Match, Value, Opportunity — that control how the composite score is calculated. They always sum to 100. Drag one slider and the others adjust automatically.

Use cases:

  • Investor / flip buyer: lower Match, raise Opportunity — surface properties with high seller motivation regardless of whether they perfectly fit your criteria.
  • Value-focused buyer: raise Value — prioritize properties priced well below neighborhood average.
  • Criteria-strict buyer: raise Match to 70+ — only see properties that actually check your boxes.

Weight changes re-rank results in real time without triggering a new Zillow search.

Session Learning

When you like or dislike properties during a session, FindRE tracks which score dimensions were high in properties you liked. After 3+ interactions, it subtly adjusts the effective weights to favor the profile of liked properties.

This is soft learning — adjustments are at most ±5 points per dimension, and they normalize to 100. The manual sliders override session learning. Session state is in-memory and does not persist between browser sessions.

Demo Mode

If no Zillow API key is configured, FindRE uses a built-in set of seeded San Diego-area listings. Demo mode is indicated in the status bar by Demo Data (instead of Zillow Live).

Everything works in demo mode: scoring, AI explanations, map, filters, session learning. It's useful for trying the UI before committing to a Zillow API key.

When the Zillow quota is exhausted mid-session, the app automatically falls back to demo data and shows Quota Exhausted (Demo) in the status bar.

Zillow API Key

FindRE uses the zillo-realtime-scraper API on RapidAPI. The key is never hardcoded in source — it's passed via environment variable or a runtime POST endpoint.

Option A: Environment variable

# .env (at project root — never committed to git)
ZILLOW_API_KEY=your_rapidapi_key_here

Option B: Runtime POST (no .env needed)

curl -X POST http://localhost:8000/api/config/zillow-key \
  -H "Content-Type: application/json" \
  -d '{"key": "your_rapidapi_key_here"}'

The key is stored in memory only — it is not written to disk and does not survive a server restart.

Quota

Each FindRE search uses approximately 1 list call + up to 10 detail calls. Monitor your usage at rapidapi.com. The free tier typically allows a few hundred calls per month — enough for regular personal use.

Ollama Setup

FindRE uses Ollama to run a local LLM for intent extraction and property explanations. No data is sent to any cloud AI service.

# Install Ollama (macOS)
brew install ollama

# Or download from https://ollama.com

# Start the server
ollama serve

# Pull the model (one-time, ~20 GB)
ollama pull qwen3.6:35b

FindRE detects Ollama automatically at startup. The status bar shows AI Ready + the model name when connected. If Ollama is not running, all features still work with rule-based fallbacks — the only difference is that intent extraction relies on simple heuristics and only the #1 result gets an AI explanation.

Model requirement: All Ollama calls pass "think": false in the request. This is required for reasoning models like qwen3.6:35b — without it, the model returns an empty response. If you switch to a different model, verify it supports this parameter.

REST API

The backend exposes a JSON REST API at http://localhost:8000/api. All endpoints below accept and return application/json.

Method + PathDescription
GET /api/statusReturns Ollama availability, Zillow configuration status, quota state
POST /api/chatSend a chat message; returns response + partial preferences + whether search is triggered
POST /api/searchRun a search given fully-formed preferences + search area + weights
POST /api/interactRecord a like/dislike on a property for session learning
POST /api/config/zillow-keySet the Zillow API key at runtime (in-memory, not persisted)
GET /docsThis user guide

Auto-generated interactive API docs (Swagger) are available at http://localhost:8000/api/docs when the server is running — but that route is reserved by FastAPI's built-in docs. Use http://localhost:8000/openapi.json for the raw schema.

FAQ

Why is the search slow?

Each search makes 1–10 calls to the Zillow API (network latency) and then runs the #1 result through qwen3.6:35b for an AI explanation (~25–35 seconds on a modern machine). The other ~20 results get rule-based explanations instantly. Total time is typically 30–45 seconds for a live search, 5–10 seconds in demo mode.

Why did my zip code land in the wrong city?

Nominatim (the geocoder) searches globally. Some US zip codes match postal districts in other countries. FindRE appends ", USA" to all five-digit zip queries to prevent this. If you get a wrong location, try using the city name instead of the zip.

Why do some properties show "lot size unverified"?

Zillow doesn't always include lot size in its API response. When lot size is absent, FindRE flags it in the explanation rather than assuming the property fails your lot requirement. Always verify lot size with the listing agent or county records.

Can I search outside San Diego?

Yes — the app works anywhere Zillow has data. Just provide any US city or zip code. The hardcoded fallback coordinates in the geocoder are San Diego-area specific, but if Nominatim can resolve your city (it usually can), the search works normally.

Where is my data stored?

Property search results are cached in a local SQLite database at backend/data/findre.db to avoid redundant Zillow API calls. This file is excluded from git. No data is sent anywhere except to the Zillow RapidAPI endpoint and (for AI features) to your local Ollama instance.

Can I run this in production?

FindRE is designed as a local personal tool. The FastAPI server has no authentication. Do not expose port 8000 to the internet without adding auth middleware.
